[JS] Prototype ES5

Wednesday, February 19, 2020

Bài này mình sẽ trình bày khá dài để chúng ta có cái nhìn tổng quát thay vì chỉ hiểu cách dùng của nó

Object và prototype: Một vòng trái đất!

Khi ta đi lên từ một số ngôn ngữ như C, C++, ... ta biết tầm quan trọng của OOP như thế nào. Nhưng còn trong Javascript, không phải hầu như mọi thứ đều là object rồi sao? Prototype làm gì? Class làm gì?

Đây là ví dụ mình hay lấy để dạy học viên của mình:

const john = {
  height: 170,
  weight: 300,
  calcBMI: function() {
    return this.height ** 2 / this.weight;
  },
};

Giờ chúng ta muốn tạo thêm một số object để lưu thêm người khác: Mark, Marry.

const mark = {
  height: 170,
  weight: 299,
  calcBMI: function() {
    return this.height ** 2 / this.weight;
  },
};
const marry = {
  height: 150,
  weight: 80, // giống đứa bạn tui
  calcBMI: function() {
    return this.height ** 2 / this.weight;
  },
};

Có lẽ bạn thấy sự bất tiện ở đây đúng không? Tạo lui tới hoài vậy, nào là lặp code, nào là không đảm bảo được cấu trúc dữ liệu của các đối tượng sẽ giống nhau.

Vậy nên làm một hàm tạo object vậy:

(Solution 1)

function personGenerator(height, weight) {
  const object = {
    height: height,
    weight: weight,
    calcBMI: function() {
      return this.height ** 2 / this.weight;
    },
  };
  return object;
}
// Và bây giờ dùng nó thôi
const john = personGenerator(170, 300);
const mark = personGenerator(170, 299);
const marry = personGenerator(150, 80);

Có vẻ cũng pơ phệch rồi đấy, nhưng nếu bạn để ý tí thì chúng ta có thể thấy hàm calcBMI trong mỗi object hoàn toàn giống nhau và sẽ bị tạo mới cùng với mỗi object, tức là nếu tạo 100 person sẽ có 100 hàm calcBMI y đúc nhau được tạo ra, thật sự lãng phí bộ nhớ. (Nếu bạn không chắc thì có thể kiểm tra john.calcBMI !== mark.calcBMI --> true).

Hừm, phải động não để sử dụng trí tuệ siêu đẳng của chúng ta, dùng các thuật cao cấp cùng với AI các kiểu, chúng ta ra:

(Solution 2)

// Vì chúng ta có thể có rất nhiều hàm cho person
const functionStore = {
  calcBMI: function() {
    return this.height ** 2 / this.weight;
  },
  eatLikePig: function() {
    console.log('Măm măm, ủn ỉn');
  },
};
function personGenerator(height, weight) {
  const object = {
    height: height,
    weight: weight,
    calcBMI: functionStore.calcBMI, // nhớ dòng này nha, tí nhắc lại
    eat: functionStore.eatLikePig,
  };
  return object;
}
const john = personGenerator(170, 300);
const mark = personGenerator(170, 299);
const marry = personGenerator(150, 80);
// john.calcBMI === mark.calcBMI --> true

Perfect! Nhưng mình sẽ chia sẻ cho bạn cái này: Object.create();

Thay vì chúng ta phải tự gán bằng tay kiểu như chỗ comment phía trên, có cách nào để "link" tự động tới functionStore hay không? Cũng là đoạn code trên, nhưng chúng ta có chỉnh sửa một tí:

(Solution 3)

function personGenerator(height, weight) {
  const object = Object.create(functionStore);
  object.height = height;
  object.weight = weight;
  return object;
}
const john = personGenerator(170, 300);
const mark = personGenerator(170, 299);
const marry = personGenerator(150, 80);
// john.calcBMI === mark.calcBMI --> true

Chuyện gì đã xảy ra? Object.create đã tạo một object rỗng kèm thuộc tính __proto__ liên kết tới functionStore. Và khi ta dùng john.calcBMI, JS sẽ nhìn vào object john xem có calcBMI hay không, nếu không có, nó sẽ tìm xuống __proto__ để vào tìm tiếp trong cái thằng nó đã liên kết đó (tức là trong object functionStore).

Một bật mí nhỏ, khi chúng ta tạo object kiểu này const ob = {}; thì __proto__ của object vừa tạo đó sẽ được liên kết với Object.prototype (Object là thằng object trùm cuối JS đó). Vì vậy, nếu ta gọi ob.toString(), đầu tiên nó sẽ tìm trong ob xem có hàm toString hay không, rõ ràng là không, sau đó nó tìm xuống __proto__ (hay chính là Object.prototype) để tìm, và ồ hố, có hàm toString. Việc đi tìm thằng này xuống thằng khác vậy, người ta gọi là: prototype chain

Done! Vậy là chúng ta có solution cuối cùng, khá pơ phệch. Tuy nhiên, thay vì làm tay như vậy, JS có một offer tốt hơn: new và constructor function

Nội dung chính

Sau một hồi tự làm tay, chúng ta đã có những yêu cầu cho một hàm tạo (constructor function: như một khung sườn, bản mẫu của các object):

  1. Hàm phải trả ra một object mới
  2. Các object có chung cấu trúc
  3. Các object có một functionStore chung (1 kho các hàm dùng chung, thay vì mỗi object có những hàm riêng giống y hệt của nhau)

Và cú pháp của hàm tạo đó là:

function Person(height, weight) {
  this.height = height;
  this.weight = weight;
}
const john = new Person(170, 300);

Giải thích một số thứ của hàm tạo cơ bản này:

  1. Tại sao đặt tên in hoa? Đó là convention mặc định vậy, thích đặt hoa thường gì cũng được. Chúng ta cũng có thể nghĩ rằng: Đặt tên in hoa để khi dùng chúng ta biết nó là một hàm tạo, phải dùng cùng toán tử new.
  2. Toán tử new có chức năng gì ở đây? Khá là dài dòng:
  • Đầu tiên, nó tạo ra một object rỗng, mới tinh, chúng ta tạm gọi nó là newObject
  • Sau đó nó truyền newObject này như một context vào hàm tạo, để this trong hàm tạo chính là object này. (Có thể nói thao tác này tương tự như Person.call(newObject, ...) hoặc Person.apply(newObject, [...]) vậy)
  • Hàm Person được gọi, this.height = height cũng tương đương với newObject.height = height
  • Cuối cùng, nếu hàm không return một object nào, thì toán tử new sẽ mặc định trả về newObject. Có nghĩa là:
john = {
  height: 170,
  weight: 300,
};
  • Nếu hàm tạo có return một loại object nào đó, ví dụ:
function Person(height, weight) {
   ...
   return { bullShit: true };
 }

thì john sẽ là { bullShit: true }

Vậy là với hàm tạo cú pháp này, chúng ta giải quyết được 2/3 yêu cầu ban đầu. Còn yêu cầu thứ 3, làm sao để các object tạo ra có chung một functionStore.

Có thể bạn đã biết hoặc chưa biết, function cũng là một loại object. Do đó, thay vì tạo ra 1 object functionStore như ở phần 1 phía trên, JS tận dụng ngay chính hàm tạo để lưu trữ các hàm dùng chung đó: Person.prototype sẽ thay cho functionStore của chúng ta (Chỉ có object mới có cú pháp object.property chứ).

Và toán tử new một lần nữa giúp chúng ta liên kết __proto__ của newObject vào Person.prototype.

Việc tạo một object như thế này đồng nghĩa chúng ta đang tạo ra một loại dữ liệu con của loại "object". Bởi vậy, khi bạn log nó ra, bạn sẽ thấy kiểu Person ngay thôi.

Tóm tắt ngắn: Khi chúng ta new một constructor function => tạo 1 object mới, đưa object đó trở thành this bên trong function đó, trả về object cho vế trái, liên kết __proto__ của object này tới Function.prototype (với Function là tên của constructor function).

Để bổ sung các phương thức dùng chung cho các object được tạo ra từ hàm tạo, chúng ta thêm bình thường vậy:

Person.prototype.sleep = function() {
  console.log("I'll go to bed now");
};
Person.prototype.calcBMI = function() {
    return this.height ** 2 / this.weight;
};
Person.prototype.eatLikePig = function() {
    console.log('Măm măm, ủn ỉn');
};
const john = new Person(...);
john.sleep(); // I'll go to bed now
// Không phải như này
Person.prototype = {
  sleep: function() {
    // ...
  },
}; // Việc này sẽ làm override nguyên object prototype và chúng ta sẽ mất các phương thức trước đó

Kế thừa

Giờ chúng ta muốn tạo ra 1 loại object là kiểu con của Person, có đầy đủ các thuộc tính và phương thức của Person, ngoài ra sẽ thêm một vài nữa. Ví dụ: Singer!

function Singer(name, weight, isBeautiful) {
  // Some code
}

Nhưng rõ ràng chúng ta không thể viết y hệt Person lại được, ngoài vấn đề lặp code còn một số vấn đề khác như sau này Person thêm thuộc tính và phương thức, chẳng lẽ lại copy bỏ vào tiếp? Do đó, chúng ta phải làm sao để Singer kế thừa được Person mà không phải lặp code lần nữa.

function Singer(name, height, weight, isBeautiful) {
  Person.call(this, height, weight); // 1
  this.name = name; // 2
  this.isBeautiful = isBeautiful; // 3
}
const chiPu = new Singer('Chi Pu', 163, 50, true);

Chà chà, có call à, hừm, khó hiểu ta.

Nếu trong bài trước bạn đã nắm rõ call hoạt động như thế nào, thì hãy thử chạy mệnh đề trên xem.

Mình sẽ giải thích cách chạy nó như thế nào:

  1. Khi new Singer(...), chúng ta sẽ có một object mới được tạo ra, hàm Singer được gọi với this chính là object mới này (Mình sẽ gọi object mới này là chiPu luôn)
  2. Ở dòng code thứ 1, hàm Person sẽ được gọi như hàm bình thường (không dùng với new), bởi vậy this trong Person chính là Window, chúng ta không muốn điều đó, this mong muốn phải là chiPu. Vì vậy, chúng ta sử dụng call gọi hàm Person để truyền context chiPu vào hàm.
  3. Sau khi chạy xong hàm Person, chiPu đã có 2 thuộc tính height và weight, chúng ta gán tiếp cho 2 thuộc tính name và isBeautiful còn lại.
  4. Cuối cùng, object chiPu được ném ra ngoài cho biến chiPu lưu.
  5. __proto__ của chiPu sẽ được link tới Singer.prototype. Done.

Hừm, còn các phương thức của Person như calcBMI, eat, sleep sẽ nằm đâu.

Bởi vì chúng ta chỉ gọi hàm Person như một hàm bình thường, bởi vậy nãy giờ chưa đụng gì tới prototype của Person.

Công đoạn cuối cùng, chúng ta làm cách nào đó để link tới prototype của Person:

Singer.prototype = Object.create(Person.prototype);

Bằng cách này, chúng ta gán cho prototype của Singer là một object rỗng, có __proto__ dẫn tới Person.prototype (Như cách hoạt động của Solution 3 phần 1)

// Thêm phương thức cho Singer
Singer.prototype.sing = function() {
  console.log('Con gà gáy o ò ó o sáng rồi ai ơi!');
};
chiPu.sing(); // Con gà gáy o ò ó o sáng rồi ai ơi!
chiPu.eat(); // Măm măm, ủn ỉn

Khi ta gọi: chiPu.sing(), đầu tiên JS tìm hàm sing trong object chiPu -> không có, tìm xuống __proto__ dẫn tới Singer.prototype và à há, tìm thấy hàm sing.

Khi ta gọi: chiPu.eat(), đầu tiên JS tìm hàm eat trong object chiPu -> không có, tìm xuống __proto__ dẫn tới Singer.prototype và à há, cũng không tìm thấy hàm eat, mò xuống __proto__ để chạy qua Person.prototype và lần này tìm được.
